# Pitfalls, red herrings, and ceilings

Every item here either looked like a fix and broke something, looked like a problem and was not, or is a hard cap no app-level change can move. Check this file before applying the playbook mechanically.

## Fixes that break things

- **Viewport-gating text splits without a same-commit reveal guarantee.** In one pass, deferring split-text work until elements near the viewport cut TBT 610ms to 280ms, then produced CLS 0.5: splitting changed section heights, and any post-paint height change shifts everything below it (with a Lenis-style scroll container, the document-end sticky footer is always "in viewport" for CLS purposes). If splitting changes layout, all splits must run in the hydration commit, before the first post-hydration paint; accept the TBT or reduce instance count. If splitting is layout-neutral, the deferred+chunked approach works, but only with the same-commit reveal guarantee (see `lcp.md`, text-splitting section).
- **Removing `next/dynamic` wrappers to consolidate chunks.** The `loading:` skeletons are what users see during client-side navigation; inlining the import removes them. Only remove `dynamic()` when there is no loading state or the component never participates in a navigation transition.
- **Preloading with mismatched CORS mode.** WebGL textures are fetched with `crossOrigin="anonymous"`, and asset CDNs (e.g. Sanity's) enforce the project CORS allowlist on those requests: a plain `<img>` load succeeds while the same URL 403s as a CORS request. Any prefetch of an image later used as a texture must use the same crossOrigin mode, and every dev port must be in the allowlist.
- **Fixing `target-size` / `color-contrast` mechanically.** Small decorative controls and low-opacity decorative labels are design decisions, and "fixing" them changes pixels. Document the option (24px hit areas, higher alpha) and get design sign-off; accept an occasional 96 otherwise.
- **Sharp poster frames instead of LQIP for videos.** Would create a real LCP candidate and improve perceived load, but blur-up vs sharp-frame is a visible difference. Design decision.
- **`experimental.turbopackTreeShaking: true` breaks Sanity Studio.** 526 module-not-found errors inside `sanity`'s lazy chunks at build time (Next 16.2 / Turbopack). The other useful flags (`turbopackScopeHoisting`, `turbopackRemoveUnusedExports`, nested async chunking) are already default in production builds.
- **SSR-visible text without an opaque cover.** Painting content under an intro overlay is safe only because the opaque overlay is in the SSR HTML. For scroll-reveal content with no cover, SSR-visible text flashes before hiding; use the `opacity: 0.001` + `inert` approach instead.

## Red herrings

- **The core-js polyfill chunk.** A ~110KB uncompressed chunk full of core-js appears in the HTML, but its script tag has `noModule`: modern browsers never download or execute it. Confirm via `network-requests` before "fixing" it.
- **Chunk-marker greps.** A chunk containing a component's name may hold only the lazy import reference. Turbopack merges modules from unrelated lazy groups into shared commons chunks, and `next/dynamic` does not guarantee exclusion from them. Only source-map attribution tells the truth (`measurement.md`).
- **A split-text library doubling textContent.** Some libraries (e.g. `@activetheory/split-text`) inject their own inline-styled `.sr-only` copy of the original text into every split host. Doubled text on split elements is expected and correct; do not "fix" it.
- **axe contrast failures on visible `aria-hidden` decoration.** axe checks color contrast on visible `aria-hidden` text (e.g. intro overlay labels caught mid-animation), and gets *more* likely to catch them as the page gets faster. WCAG exempts pure decoration but axe cannot know that. Bump the alpha with design sign-off or accept a 96 that occasionally reads 100.
- **`modern-http` insight on localhost.** `next start` is HTTP/1.1; production is HTTP/2 + CDN. Local complaint is expected and not actionable.
- **A CLS flake blaming the last diff.** With a document-end `position: sticky; bottom: 0` footer behind the page (scroll-reveal pattern): (a) any late footer geometry change (fonts settling, streamed HTML, hydration) registers as a large CLS entry in whichever run loses the timing race, so CLS flakes between 0 and ~0.5 across identical builds; (b) axe computes the footer's text contrast against the page background covering it at scroll-top and fails `color-contrast`. Neither is a regression of the last change; both need footer geometry final at first paint (fixed row heights independent of font metrics) or a rethought reveal structure. Diagnose with the buffered layout-shift observer (`measurement.md`), not by reverting diffs.
- **One a11y pattern applied to all content.** `role="text"` + `aria-label` host + `aria-hidden` lines is valid only for plain text; with links inside it hides focusable elements from assistive tech and fails three audits at once. Branch on content (`lcp.md`, Family A).

## Hard ceilings (document, do not chase)

- **Framework floor.** ~500KB raw / ~130-140KB gz of framework evaluates on every page of a Next app; a single eval+hydration task of ~550ms (throttled) exists even on the leanest page. Below roughly 240KB gz initial JS, localhost-simulated LCP and TBT stop responding to app-level changes.
- **Lantern text-LCP floor.** Once observed LCP is at first paint, simulated LCP tracks total initial JS (see `measurement.md`). Framework + animation library + app code can cap simulated mobile LCP around ~4s and the mobile score in the 60s-80s while real-user LCP is 100-300ms. Do the byte math, write the cap down, and stop.
- **Always-animating WebGL / marquee backgrounds.** three.js eval lands in TBT no matter when it loads (deferring just moves the long task within the trace), and Speed Index never settles because the page is never "visually complete" (~9s SI measured under devtools throttling). Only a cheaper visual (CSS / 2D canvas) changes this; that is a design decision.
- **Autoplaying video walls.** Decode and network for dozens of in-view videos dominate TBT and bandwidth. `preload="none"` + IntersectionObserver-driven `play()` is the mitigation; beyond that it is a design tradeoff.
- **Intentional hero effects.** When the design requires a WebGL hero plus animated text reveal, optimize around them (idle-mount the canvas, paint the text pre-reveal) and document the residual cost. Do not gate the feature off to buy points.
